适配器模式

    • 1.定义
      也称变压器模式。将一个类的接口变换成客户端所期待的另一种接口，从而使原本接口不匹配而无法在一起工作的两个类能够在一起工作。

    • 2.角色

    • 3.优缺点

      优点:让两个没有任何关系的类能一起运行
      增加了类的透明性
      提供类的复用性
      增强代码灵活性

    • 4.应用场景

      将不符合系统接口的类进行转化

责任链模式

2019-04-01

  • 概述

    • 1.定义
      使多个对象都有机会处理请求，从而避免请求的发送者和接收者之间的耦合关系。
      这些对象连成一条链，并沿着这条链传递该请求，直到有对象处理它为止

    • 2.角色

    • 3.优缺点

      优点: 将请求和处理分开，请求者不知道谁处理的，处理者也不指定请求的情况
      提供系统的灵活性
      缺点: 降低程序的性能，当链很长的时候，性能大幅下降
      不易于调试

    • 4.应用场景

      一个请求需要一系列的处理工作
      业务流的处理
      对文件系统进行补充扩展

访问者模式

2019-04-01

  • 概述

    • 1.定义
      封装一些作用于某种数据结构中的各元素操作，可以在不改变数据结构的前提下定义作用于这些元素的新操作

    • 2.角色

    • 3.优缺点

      优点:增加新操作变得很容易，只需要增加新的访问类
      将有关行为集中到一个访问者对象中
      跨多个类进行访问
      累积状态
      缺点:增加新的元素类变得困难，必须在抽象访问者角色中增加新的抽象操作
      破坏封装，每个对象被访问，就需要暴露自己的操作和内部状态
      违背了依赖倒置原则，访问者依赖的是具体的元素而不是抽象的元素

    • 4.应用场景

      一个对象包含很多类对象，迭代器不能实现的情况下
      需要对一个对象结果中的对象进行不同的且不相关的操作时
      遍历多个不同的对象，而迭代器是遍历多个相同的对象

组合模式

2019-04-01

  • 概述

    • 1.定义
      将对象组合成树形结构以表示“部分-整体”的层次结构，使得用户对单个对象或者整个对象使用具有一致性。

    • 4.应用场景

      树形菜单、文件管理

  • epoll

          Linux 2.4之前主要使用select和poll.在Linux 2.6的时候正式引入epoll.其是高性能网络服务器的必备技术。select和epoll都是使用一个线程来处理所有连接的线程，但是select错误预估这所有的连接中只有比较少的活跃连接。显然当连接数量达到数以万计的时候而返回的活跃连接数只有几百个的时候，select就比较低效了，因为其要遍历数以万计的连接数。其次内核中的select采用轮询的方法，每次检测所有FD_SET的句柄，而句柄是存在操作系统的限制的，一般只有1024个。poll在这方面取消了最大监控文件描述符的限制，但是没有解决select的存在问题。
          epoll的三个核心方法是1.新建epoll描述符(epoll_create()),2.添加事件(epoll_ctl()),3.返回活跃的连接(epoll_wait())。epoll三大关键要素是1.mmap这是将用户空间的一块虚拟地址和内核的一块虚拟地址同时映射到一块物理地址上，使得这块物理内存对内涵和对用户均可可见，减少用户太和内核态的数据交换。2.红黑树,存储epoll所监听的套接字，当使用epoll_ctl添加或者删除一个套接字的时候都会去红黑树上处理。3.链表，rdlist。

          工作原理

    • epoll_wait调用ep_poll,当rdlist为空,挂起当前进程，直到rdlist不空时候唤醒进程
    • 文件fd状态改变,导致相应fd上的回调函数ep_poll_callback触发
    • ep_poll_callback将相应fd对应的epitem加入rdlist,导致rdlist不空，唤醒进程,epoll_wait继续执行
    • ep_events_transfer将rdlist中的epitem拷贝到txlist中，清空rdlist
    • ep_send_events，扫描txlist中的每个epitem,调用其关联fd对应的poll方法,将取得的events和相应的fd发送到用户空间中,epoll_wait函数返回
            LT和ET模式。在读模式下，LT在buffer中有数据的时候就认为buffer可读，返回读就绪。ET只在buffer容量再次改变的时候才认为buffer可读，返回读就绪。上面是对于ADD IN事件来说，对于MOD IN事件则会直接返回读就绪。在写模式下，LT在buffr有写空间的时候就会返回写就绪，ET则在buffer写的数据发生变化(比如读取),返回写就绪。对于MOD IN事件睡直接返回写就绪。但是epoll会存在一个问题，也就是NT模式是如何保证用户要求写的数据都写完或者读的数据都读完，不好出现数据丢失问题。

相关

  • 内存映射文件

          操作系统可以利用虚拟内存实现一个文件或者文件部分映射到内存中，然后这个文件可以当作内存数组一样地访问，这比传统的文件操作快很多。
          首先从文件中获得一个通道，通道是用于磁盘文件的一种抽象，使得可以访问如内存映射，文件加锁机制以及快速在文件间数据传递操作。FileChannel channel=FileChannel.open(path,options);。然后使用channel的map方法获得一个ByteBuffer,其支持映射模式有MapMode.READ_ONLY,MapMode.READ_WRITE,MapMode.PRIVATE。后者是表示修改对缓冲区可见私有，不会传播到文件中。这样就可以使用ByteBuffer类来进行读写数据。缓冲区支持顺序和随机数据访问，通过get和put方法来操作为止。
          关于缓冲区的操作有。遍历缓冲区数据:while(buffer.hasRemaining()) byte b=buffer.get();或者for(int i=0;i\<buffer.limit();i++) byte b=buffer.get(i);以上是读取单个字节的遍历。对于读写字节数组，则需要使用get(byte[] bytes)或者get(byte[] bytes,int offset,int length)。put操作类似。除了可以读写字节，还能读写整数，浮点数等。

  • 数据结构

          Buffer包括ByteBuffer,CharBuffer,DoubleBuffer,IntBuffer,LongBuffer和ShortBuffer。每个缓冲区有一个容量(buffer.size()),一个读写位置下次读取或者获取值将在此进行(buffer.get()),一个界限超过它进行读写是没有意义的(buffer.limit()),一个可选标记用于重复一个读写操作(buffer.rewind()将读写位置复位到0，保持界限不变;buffer.mark()将编辑设置到读写位置返回;buffer.reset()将位置设置到标记处).
          调用Buffer的flip方法将界限设置到当前位置，并且读写位置复位到0。调用Buffer的clear方法使得缓冲区为下一次写做好准备,clear将读写位置复位到0，界限复位到容量。使用Buffer可以替代随机访问文件。
    0<=标记<=读写位置<=界限<=容量

  • 序列化

          对于保存数据到文件中，如果是基本类型实验DataOutputStream即可，但是对于对象类型则需要实验ObjectOutputStream其继承了DataOutputStream类，实现了OutputStream和ObjectStreamContants接口。对于序列化的类需要实现Serializable接口。一般来说，实现了该接口需要提供SerialVersionUID字段，如果不显示指定则由程序自行计算。如果序列化前的类与序列化后的类在版本号SerialVersionUID不一致的话，会报异常。当版本SerialVersionUID一致的时候，但此时有多个样本，每个样本存在新旧之分，比如说新样本中的字段多。这样当从旧样本中恢复到新样本，那些新加的字段会被赋空。反之则旧样本中没有的字段会被删除掉不显示。可以通过在命令行中实验serialver命令查看一个全类名的类对应的SerialVersionUID。或者使用serialver -show在图形化界面中输入全类名，来显示。当修改文件后需要重启该图形界面。
          序列化的过程是，在写入到文件的时候，会主动调用ObjectOutputStream流中的writeObject()方法，该方法会讲类中的实例且非transient类型的进行保存到文件中，在反序列化的时候进行读取文件中的数据到对象中。在反序列化的时候会执行对象的构造函数，因此反序列化的对象和原来对象是不一样的。但是却是深拷贝，因此在不考虑性能开销的情况，很适合作为对象的clone方法使用。如果想要保证序列化前后的对象是一样的，也就是地址也完全一致，那就需要在类中定义一个方法readResolver，该方法会成为readObject的返回值。
          对于自己需要实现序列化逻辑或者想实现transient数据的实例化，可以在类中加上自定义的writeObject和readObject方法，这些方法都是private修饰的，在程序中会通过反射来获取到，对于上面的readResolver最好也是private的。这样在writeObject中首先调用oos.defaultWriteObject方法来对非transient的实例数据进行实例化，然后只需要考虑transient的实例化即可。读取的适合类似，使用ois.defaultReadObject来读取即可。对于数据写入和读取的顺序要保持一致性。除了使用Serializable接口外，还可以使用Externalizable接口，该接口必须要实现两个方法readExternal和writeExternal，在方法中完全自定义序列化，包括对基类的数据进行序列化等等，方法是public的。
          每个对象都有一个序列号，相同对象重复出现将被存储为对这个对象的序列号的引用，对于transient类型的数据可能是冗余或者是为了安全或者是需要自定义序列化等
